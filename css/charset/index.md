## Кратко

`@charset` задаёт кодировку текущего CSS‑файла. Должна быть первой инструкцией в файле (после возможного BOM). В совремённой практике с UTF‑8 почти не требуется: кодировку лучше указывать HTTP‑заголовком или через `<meta charset>` для HTML.

## Пример

```css
@charset "UTF-8";

body {
  font-family: system-ui, sans-serif;
}
```

Проигнорируется, если перед директивой будет комментарий:

```css
/* Я комментарий */
@charset "UTF-8"; /* будет проигнорировано */
```

## Как пишется

- Записывается только на верхнем уровне и только первой строкой файла.
- Строковое значение в кавычках, обычно `"UTF-8"`.
- Допускается один раз на файл. Если встречается несколько, учитывается только первая.
- Должна предшествовать любым `@import` и другим правилам.
- в HTML-документе кодировка определяется документом, поэтому @charset внутри [`<style>`](/html/style/) игнорируется.

## Как понять

Раньше браузеры не всегда могли корректно определить кодировку CSS-файла без явного указания. Парсеру CSS нужна корректная кодировка, чтобы верно прочитать не ASCII‑символы (например, селекторы, комментарии, литералы). Исторически для этого использовали `@charset`. Сейчас повсеместно применяют UTF‑8 и задают кодировку уровнем выше:

- HTTP‑заголовком `Content-Type: text/css; charset=utf-8`;
- BOM (U+FEFF) в начале CSS‑файла;
- `<meta charset="utf-8">` для HTML (влияет на встроенные стили).

<aside>

Любой символ перед `@charset` делает его невалидным

</aside>

## Подсказки

💡 Предпочитайте UTF‑8 и настройку кодировки на уровне сервера/сборки. `@charset` используйте только при необходимости и строго первой строкой.
💡 Перед `@charset` не должно быть ни пробелов, ни комментариев, ни пустых строк.
💡 Если файл импортируется через [`@import`](/css/import/), его собственный `@charset` тоже обязан стоять самым первым в импортируемом файле.
💡 Никогда не смешивайте разные кодировки в одном проекте — это источник трудноуловимых ошибок.
